스크립트 API 매뉴얼¶

Lucy Studio 스크립트 환경(자바스크립트 기반)에서 사용할 수 있는 엔진 제공 API(전역 객체, 메서드, 라이프사이클)의 상세 명세입니다.

1. 스크립트 API 개요¶

Lucy Studio Script Engine은 폼(Form) 로딩 시점에 엔진의 핵심 객체들을 스크립트 컨텍스트에 등록(registerObject) 합니다. 이 과정을 통해 개발자는 별도의 선언 없이 전역 객체를 통해 API를 즉시 호출할 수 있습니다.

1.1. 주요 전역 객체¶

스크립트 내에서 즉시 사용 가능한 전역 객체는 다음과 같습니다.

  • $vm: 데이터 및 통신 관리
  • $form: 화면 및 인터랙션 제어
  • $page: 페이지 상태 및 스토어 관리
  • $app: 앱 전역 정보 및 네비게이션
  • $log: 디버깅 로그 출력

1.2. 호출 예시 (Usage)¶

2. 스크립트 로딩과 실행 흐름¶

스크립트는 엔진의 처리 메커니즘에 따라 특정 순서로 로드되고 실행됩니다. 특히 외부 모듈 로드와 초기화 시점을 정확히 파악하는 것이 중요합니다.

2.1. require() 선로딩 (Module Load)¶

스크립트 파일 최상단에 require()를 선언하면, 엔진은 본문 스크립트를 로드하기 전 해당 모듈들을 먼저 로드합니다.

  • 문법: require('모듈경로'); 또는 require("모듈경로"); (홑따옴표와 쌍따옴표 모두 지원)
  • 확장자 처리: 엔진 내부에서 자동으로 .js 확장자를 붙여 로드합니다.
    • 예: require('common/util') 호출 시 실제로는 common/util.js 파일을 로드함

2.2. onStart() 실행 (Initialization)¶

폼이 초기화되면 onStart(linkArg) 가 호출됩니다. linkArg 는 “이전 페이지/폼에서 전달한 argument”입니다.

3. 라이프사이클 콜백 (Lifecycle Callbacks)¶

아래 함수들은 스크립트 내에 선언해 두면, 특정 이벤트 발생 시점에 엔진이 **자동으로 호출**합니다.

3.1. 기본 라이프사이클¶

함수명 호출 시점 설명
onStart(linkArg) 폼 시작 시 초기화 시점 (1회 호출)
onActivate() 활성화 시 화면이 전면에 나타나 활성화될 때
onInactivate() 비활성화 시 화면이 뒤로 가거나 비활성화될 때
onClose() 폼 종료 시 폼이 닫힐 때

3.2. 메시지 및 데이터 통신 관련 콜백¶

데이터의 요청부터 수신, 반영까지의 세부 시점을 제어할 수 있는 콜백입니다.

분류 함수명 설명
메시지 onMessage(name, data) 폼 간 전달된 메시지 수신 시
요청 제어 onRequestBefore(provInstName) Provider 요청(API 호출) 직전
onRequestAfter(provInstName) Provider 요청(API 호출) 직후
오류 처리 onReceiveError(provInstName, code, message) 데이터 수신 중 에러 발생 시
데이터 반영 onResponseUpdateBefore(contentsDSName, provInstName) 응답 데이터를 데이터셋에 반영하기 직전 (정렬/전처리)
onResponseUpdateAfter(contentsDSName, provInstName) 데이터셋 반영 직후 (바인딩/후처리)
Push 데이터 onPushUpdateBefore(contentsDSName, provInstName) 실시간 Push 데이터 갱신 전
onPushUpdateAfter(contentsDSName, provInstName) 실시간 Push 데이터 갱신 후
기타 통신 onDataMessage(provInstName, code, message) 데이터 메시지 수신 시
onReceiveDone(provInstName, isSuccess) 데이터 수신 프로세스 완료 시 (성공 여부 포함)

4. 전역 객체(Globals) 한눈에 보기¶

Lucy Studio 스크립트 환경에서 가장 빈번하게 사용되는 핵심 전역 객체 목록입니다. 각 객체는 고유의 관리 영역을 가지며, 이를 통해 앱의 모든 요소를 제어할 수 있습니다.

객체명 관리 범위 및 역할
$form 현재 활성화된 폼(화면)의 제어, 팝업 호출, 메시지 송수신, 로딩 처리 및 프로퍼티 관리
$parentForm 현재 폼을 포함하고 있는 상위 폼 객체 참조
$ownerForm 팝업이나 컨테이너 소유 관계에 있는 소유자(Owner) 폼 객체 참조
$rootForm FormContainer를 기준으로 하는 최상위 폼 객체 참조
$page 페이지 범위 내의 변수, 스토어(Store), 공유 데이터 관리
$vm 데이터 요청(Provider 호출), 데이터셋(DataSet) 조작, 구독 및 페이지네이션 제어
$app 앱 전역 기능(페이지 전환, 앱 레벨 스토어/변수, 테마 설정 등) 관리
$log 개발 및 디버깅을 위한 시스템 로그 출력

5. $form API (JetRunForm)¶

$form 객체는 현재 화면의 내비게이션, 폼 간의 통신, 그리고 화면 내 유틸리티 기능을 제공하는 가장 핵심적인 API입니다.

5.1. 화면 열기/닫기 (Navigation & Popup)¶

화면 이동 및 다양한 형태의 팝업(Drawer, Sheet, Dialog)을 호출하거나 닫을 때 사용합니다.

분류 메서드(Method) 매개변수 설명
페이지 $form.openPage path, arg, route 지정된 경로의 페이지로 이동
$form.openBackPage - 이전 페이지로 돌아가기
Drawer $form.openStartDrawer path, arg, cb? 화면 왼쪽(Start)에서 슬라이드되는 Drawer 호출
$form.openEndDrawer path, arg, cb? 화면 오른쪽(End)에서 슬라이드되는 Drawer 호출
Sheet $form.openSheet path, arg, cb? 하단에서 올라오는 BottomSheet 팝업 호출
$form.openModalSheet path, arg, cb?, l, t, w, h 모달 시트 호출 (좌/상/폭/높이 지정 시 하이라이트 오버레이)
Dialog $form.openDialog path, arg, cb? 중앙 다이얼로그 팝업 호출
$form.openFullDialog path, arg, cb? 전체화면/라우트 기반 다이얼로그 호출
닫기 $form.close result 현재 폼을 닫고 부모에게 결과값 반환

5.2. 폼 간 메시지 통신 (postMessage / onMessage)¶

서로 다른 폼 간에 데이터를 송수신하는 핵심 메커니즘입니다.

메서드(Method) 매개변수 설명
$form.postMessage name, data, target? 대상에게 메시지 전송
(target 규칙: null=자기자신, "*"=브로드캐스트, "A,B"=특정 컨테이너)
$form.postMessageAll name, data 전체 하위 폼까지 메시지 전파
$form.postDelayedMessage name, data, target?, ms 지정된 지연 시간(ms) 후 메시지 전송
$form.postMessageToApp name, data, ms 다른 페이지나 앱 전역 리스너로 메시지 전송

5.3. 폼 변수, 속성 및 유틸리티¶

분류 메서드(Method) 매개변수 설명
로컬 변수 $form.getVar / setVar / clearVar name, (data) 현재 폼 내 로컬 변수 관리
프로퍼티 $form.getProperty / setProperty name, (value) 폼 프로퍼티 접근 (위젯 ID 접근 방식과 별개)
라디오 $form.getRadioGroupValue / setRadioGroupValue groupId, (val) 라디오 그룹 값 조회 및 설정
로딩 $form.showLoadingIndicator / hideLoadingIndicator timeoutSeconds 로딩 인디케이터 표시/숨김
지연 실행 $form.runDelayedScript ms, function 지정 시간 후 스크립트 지연 실행

⚠️ 주의: 프로퍼티 설정 시 구분 사항$form.setProperty는 폼 전체의 속성을 다루는 API입니다. 특정 위젯의 속성을 변경하고 싶을 때는 위젯의 고유 ID를 사용하는 widgetId.setProperty(...) 규칙을 따라야 하며, 이 둘은 서로 엄격히 구분됩니다.

6. $page API (JetRunPage) — 페이지 범위 데이터 관리¶

$page 객체는 현재 페이지 내에 존재하는 폼(Form)들 간에 데이터를 공유하거나, 페이지 수명 주기 동안 유지되어야 하는 상태 값을 관리할 때 사용합니다.

분류 메서드(Method) 매개변수 설명
페이지 변수 $page.getVar / setVar / clearVar name, (data) 현재 페이지 범위 내 변수 관리
공유 데이터 $page.getSharedData / setSharedData key, (data) 페이지 내 메모리 맵 기반 데이터 공유
저장소 $page.getStoreData / setStoreData key, (data) 페이지 단위 영속 저장소 관리

7. $app API (KAppManager) — 앱 전역 기능¶

$app 객체는 엔진 JSON 기준으로 전역 객체로 제공되며, 앱 전체의 상태, 테마, 플랫폼 정보 및 페이지 이동을 제어합니다.

분류 메서드(Method) 매개변수 설명
이동 $app.openPage / openPageReplacement / openBackPage path, arg, route? 앱 전역 페이지 이동 및 교체
공유 데이터 $app.getSharedData / setSharedData key, data 앱 전역 런타임 공유 데이터 관리
저장소 $app.getStoreData / setStoreData / removeStoreData key, (data) 앱 전역 영속 저장소 관리
전역 변수 $app.getVar / setVar / clearVar name, (data) 앱 전역 변수(AppVars) 관리
테마/환경 $app.changeTheme / isDarkTheme / isDeviceDarkTheme type 테마 설정 및 다크모드 여부 확인
플랫폼 $app.isAndroid / isIOS / isWindows / isMacOS / isLinux - 실행 중인 OS 플랫폼 확인 (boolean)
시스템 $app.operatingSystem / localeName / localHostname / packageInfo - 시스템 로케일 및 패키지 정보 조회

8. $vm API (JetVM) — 데이터 통신 및 데이터셋 제어¶

$vm은 **외부 데이터 요청(Provider) → 데이터셋(DataSet) 반영 → UI 바인딩 반응**으로 이어지는 Lucy Studio 데이터 흐름의 핵심 객체입니다.

8.1. Provider 요청¶

메서드(Method) 매개변수 설명
$vm.requestData provInstName 최초 데이터를 서버에 요청합니다.
$vm.requestNextData provInstName 다음 페이지 데이터를 요청합니다. (페이지네이션)
$vm.hasMoreNextData provInstName 다음 데이터가 존재하는지 여부를 확인하여 boolean으로 반환합니다.

💡 실무 활용 팁 리스트 스크롤이 끝에 도달했을 때 hasMoreNextData()를 체크한 후, 결과가 true일 경우에만 requestNextData()를 호출하는 패턴으로 주로 사용합니다.

8.2. 실시간/푸시 구독¶

메서드(Method) 매개변수 설명
$vm.subscribeData provInstName, func? 실시간 데이터를 구독합니다.
$vm.unsubscribeData provInstName, func? 특정 데이터 구독을 해제합니다.
$vm.cancelSubscribeData provInstName, func? 구독 취소 처리를 수행합니다.
$vm.cancelAllSubscribeData - 모든 실시간 데이터 구독을 일괄 취소합니다.

8.3. DataSet 값 조작¶

메서드(Method) 매개변수 설명
$vm.setDSValue path, jsonData 특정 데이터셋 경로에 값을 설정합니다.
$vm.getDSValue path 특정 데이터셋 경로의 값을 가져옵니다. (dynamic 반환)
$vm.insertDSValue recordPath, jsonData 레코드 경로에 데이터를 삽입합니다. (boolean 반환)
$vm.appendDSValue recordPath, jsonData 레코드 경로 끝에 데이터를 추가합니다. (boolean 반환)
$vm.deleteDSValue recordPath, count 레코드 경로에서 지정된 개수만큼 데이터를 삭제합니다. (boolean 반환)

⚠️ 데이터셋 경로(Path) 규칙 DataSet path 문자열 규칙은 프로젝트 설정(데이터셋/레코드 구조)에 따라 달라질 수 있습니다. 가장 안전한 방식은 **UI 바인딩 시 사용한 경로/필드명**을 그대로 재사용하는 것입니다.

8.4. DataSet 커서 (현재 index)¶

메서드(Method) 매개변수 설명
$vm.setDSCurrent path, current 데이터셋의 현재 인덱스(커서)를 설정합니다.
$vm.getDSCurrent path 데이터셋의 현재 인덱스를 가져옵니다. (int 반환)

8.5. JSONPath / 필터링 유틸¶

메서드(Method) 매개변수 설명
$vm.evalDSJsonPath dmPath, regPath 데이터셋 내에서 JSONPath를 사용하여 데이터를 필터링합니다.
$vm.evalProviderJsonPath recvPath, regPath 수신된 원천 데이터 내에서 JSONPath를 사용하여 필터링합니다.

8.6. 정렬¶

메서드(Method) 매개변수 설명
$vm.sortNumber dmName, key, asc 숫자를 기준으로 데이터를 정렬합니다.
$vm.sortString dmName, key, asc 문자열을 기준으로 데이터를 정렬합니다.

💡 정렬 시 주의사항 asc(오름차순) 값은 엔진 JSON 기준 문자열로 입력받으므로, 프로젝트 내에서 "true" / "false" 규칙을 통일하여 사용하는 것을 권장합니다.

8.7. 업데이트 잠금 (데이터 폭주/경합 방지)¶

메서드(Method) 매개변수 설명
$vm.isDsUpdateLock dmName 특정 데이터셋의 업데이트 잠금 상태를 확인합니다.
$vm.dsUpdateLock dmName, updateLock 데이터셋의 업데이트 잠금 여부를 설정합니다.

8.8. 기타¶

메서드(Method) 매개변수 설명
$vm.getProviderData path 수신된 원천(Provider) 데이터를 가져옵니다. (dynamic 반환)
$vm.getDataCount fieldPath 특정 필드 경로의 데이터 건수를 반환합니다. (int 반환)
$vm.loadFileData piName, fileName 파일 데이터를 로드합니다.
$vm.getPushResultIndex dmName, blockName 푸시 결과 인덱스를 리스트로 반환합니다.
$vm.requestCommand cmd, data, channel 특정 커맨드와 데이터를 지정된 채널로 요청합니다.

9. $log API (LogManager)¶

메서드 레벨 설명
$log.d(any) Debug 개발 단계에서 상세한 추적 정보나 변수 값을 확인할 때 사용
$log.i(any) Info 정상적인 로직 흐름이나 서비스 정보 확인용
$log.w(any) Warn 잠재적인 오류나 주의가 필요한 상황을 출력
$log.e(any) Error 예외 상황이나 로직 실패 등 치명적인 오류 발생 시 사용

💡 스크립트 디버깅 팁 Lucy Studio에서 스크립트 디버깅을 할 때는 **$log 호출**과 **라이프사이클 콜백(onStart, onResponseUpdateAfter 등)**을 조합하여 로그를 출력하면, 어느 시점에 데이터가 들어오고 화면이 변하는지 유용하게 추적할 수 있습니다.

10. (참고) 위젯/Object ID 기반 API 사용 시 주의사항¶

스크립트 내에서 위젯이나 데이터를 참조할 때는 명칭의 일관성이 무엇보다 중요합니다. 오류 없는 데이터 바인딩과 제어를 위해 아래의 핵심 규칙을 반드시 준수해야 합니다.

✅ 핵심 규칙 (Naming Rule)¶

“위젯 ID / Provider Instance Name / 스크립트에서 참조하는 이름”을 반드시 동일하게 유지해야 합니다.

  • 1:1 매칭: 특히 Provider 이름은 스크립트에서 사용되는 $vm.requestData("...") 내의 문자열 값과 반드시 1:1로 정확하게 매칭되어야 합니다.
  • 영향도: 이름이 서로 다를 경우 엔진이 해당 객체를 찾지 못해 데이터 요청이 실패하거나 위젯 제어가 불가능해집니다.

11. 초보자용 “가장 흔한 패턴” 예시¶

이 섹션에서는 Lucy Studio에서 스크립트를 작성할 때 가장 자주 사용되는 비즈니스 로직 패턴 6가지를 다룹니다.

11.1. 폼 시작 시 데이터 요청 및 공통 훅 (Hook)¶

폼이 로드될 때 데이터를 요청하고, 요청의 전/후 처리 및 에러 발생 시의 대응 로직을 구현하는 가장 기본적인 패턴입니다.

사용된 주요 요소 역할
onStart 폼 초기화 시점에 $vm.requestData로 데이터 요청 시작
onResponseUpdateAfter 데이터셋 반영 후 추가적인 정렬이나 계산 수행
onRequestBefore / After 로딩 인디케이터(showLoadingIndicator) 표시 및 숨김 제어
onReceiveError / Done 통신 결과(에러/완료)에 따른 후속 처리 및 로그 기록

11.2. 스크롤 끝에서 페이지네이션 (Pagination)¶

리스트나 그래프 등에서 스크롤이 끝에 도달했을 때 다음 데이터를 추가로 불러오는 패턴입니다.

11.3. 실시간/푸시 구독 (Subscribe) 패턴¶

실시간으로 변하는 데이터를 수신하기 위해 구독을 시작하고, 폼이 닫힐 때 구독을 해제하여 자원을 관리하는 패턴입니다.

11.4. 폼 간 메시지 통신 (postMessage / onMessage)¶

서로 다른 화면(폼) 간에 이벤트를 전달하고, 수신된 이벤트에 따라 로직을 수행하는 통신 패턴입니다.

11.5. 팝업 열기 및 결과 수신 (Sheet/Dialog)¶

하단 시트나 다이얼로그를 열고, 해당 팝업에서 작업이 완료된 후 결과값을 받아 처리하는 상호작용 패턴입니다.

11.6. 변수 및 데이터 저장소 (Store) 활용 패턴¶

데이터의 유효 범위에 따라 로컬 변수, 페이지 변수, 영속 저장소(Store)를 구분하여 사용하는 패턴입니다.

저장 방식 객체 및 범위 특징
폼 로컬 변수 $form.setVar 현재 폼(화면) 안에서만 유지
페이지 변수 $page.setVar 현재 페이지 전체 범위에서 유지
페이지 저장소 $page.setStoreData 페이지별로 유지되는 영속 데이터 (캐시 등)
앱 저장소 $app.setStoreData 앱 전체에서 유지되는 영속 데이터
